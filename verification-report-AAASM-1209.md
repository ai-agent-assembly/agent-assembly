**Date:** 2026-05-21 (UTC)

## Why dry-run instead of a real tag

The original verification AC said "Push a test tag `v0.0.1-rc1`". Doing that literally would fire `cargo publish -p aa-cli` against crates.io, burning the `0.0.1-rc1` version permanently — and the workspace version is `0.0.1` (no pre-release suffix), so the publish step would also fail the version-match check.

This PR replaces the test-tag path with a `workflow_dispatch` dry-run path that exercises the matrix build + archive + SHA256SUMS for all 4 platform targets without mutating crates.io or creating a GitHub Release. `cargo publish` and `softprops/action-gh-release` are gated behind `github.event_name == 'push'` so they still fire on real `v[0-9]+.[0-9]+.[0-9]+` tag pushes.

## Runner matrix change

The original matrix targeted `x86_64-apple-darwin` to **macos-13**. The first dry-run attempt ([run #26176204851](https://github.com/ai-agent-assembly/agent-assembly/actions/runs/26176204851)) sat queued for **7h32m** on macos-13 — GitHub-hosted Intel macOS capacity is severely constrained as that image winds down. The fix (commit `🐛 (ci): Cross-compile x86_64-apple-darwin on macos-14`) moves the runner to macos-14 (Apple Silicon), cross-compiles via the rustup-installed x86_64 target, and installs Rosetta 2 inline so the `aasm --version` smoke step still exercises the produced Intel binary.

### AC 4 — binary sizes in 2–8 MB

| Target | Actual | AC target |
| --- | --- | --- |
| aarch64-unknown-linux-gnu | **15 MB** | 2–8 MB |
| x86_64-unknown-linux-gnu | **15 MB** | 2–8 MB |
| aarch64-apple-darwin | **12 MB** | 2–8 MB |
| x86_64-apple-darwin | **13 MB** | 2–8 MB |

All four targets exceed the upper bound by 4–7 MB. The `[profile.release]` flags from AAASM-1206 (`opt-level="z"`, `lto=true`, `codegen-units=1`, `strip=true`, `panic="abort"`) are in place; the residual size is likely the embedded dashboard assets plus the linked-in `aa-api`/`aa-gateway` stack.

**AC 4: FAIL.** Follow-up filed as **AAASM-1683** ([BUG] aasm release binary exceeds 2–8 MB AC target). The investigation in that subtask will decide between (A) revising the AC, (B) a slim-build feature flag, or (C) moving dashboard assets out of the binary.

### AC 5 — `cargo install aasm` succeeds after crates.io publish

The `publish-crate` job is gated to `github.event_name == 'push'` (commit `🔧 (ci): Gate crates.io publish on real tag push`) and was correctly skipped on this dispatch run. Exercising this AC requires an irreversible `cargo publish` of `0.0.1` to crates.io, which is deliberately deferred until v0.0.1 is actually cut.

**AC 5: deferred.** Will be exercised by the v0.0.1 tag push.
