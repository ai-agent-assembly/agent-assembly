## v0.0.1-rc.4 — release-artifact completeness + hardening

Fourth release candidate in the v0.0.1 series (patch on the `rc` channel,
`rc.3 → rc.4`). The headline of this cut is **release-artifact completeness**:
the `aa-api-server` binary and the `aa-gateway` container image are now part of
the published release set, and a completeness gate guards against a component
silently dropping out again. Folded in are the accumulated security-hardening
sweeps and the local-mode gRPC registration surface merged since `rc.3`.

> **Not for production use.** At `0.x.y` SemVer, rc carries no API/ABI/wire-protocol
> stability commitment. `protocol/v1` is unchanged.

### Release-artifact completeness (the rc.4 driver)

- **`aa-api-server` now shipped** (AAASM-4449) — the binary is built (`-p aa-api`),
  its presence is verified, and it is packaged into the `api` component tarball;
  the `components.json` glob was widened to include `aasm-api`. Previously the
  binary was absent from the release set.
- **`aa-gateway` image published** (AAASM-4480) — a distroless `cargo-chef`
  Dockerfile is built on PR validation and pushed to GHCR on tag, alongside the
  existing `python` / `go` images.
- **Release-completeness gate** (AAASM-4456) — a script + CI job assert every
  expected component is present, so a missing artifact fails the check rather
  than shipping a partial release.

### Local-mode gRPC registration (AAASM-4447)

`aa-api` now serves the gRPC `AgentLifecycleService` on loopback in local mode,
backed by a durable SQLite registry, so an SDK-registered agent is visible over
REST without standing up a full gateway. New conformance registration-surface
contract tests assert the gateway serves the SDK gRPC registration service and
that it declares the full lifecycle.

### Analytics API + dashboard

Seven `/api/v1/analytics/*` endpoints (KPIs, cost-breakdown, action-volume,
tool-usage, approvals, policy-effectiveness, fleet-health) with the dashboard
Analytics/Costs pages wired to live data, per-panel error isolation, and
server-side windowed audit reads (AAASM-4131 / 4138 / 4142 / 4155 / 4158).

### Security — accumulated sweeps since rc.3

- **`tools: "*"` wildcard** honored across all three tool stages (allow /
  rate-limit / approval) on both single-file and cascade paths — previously
  fail-open (Epics AAASM-4149 / 4163).
- **Rate-limit bypass** fixes: anonymous shared-bucket path (AAASM-4190),
  multi-policy application, and per-tenant bucket keying (AAASM-4171 / 4173).
- **Fail-closed policy loading** on unknown top-level and nested section keys
  (AAASM-4189 / 4330); **fail-closed budgeting** on unknown LLM models plus
  org/team tier caps and ancestor monthly cap (Epic AAASM-4092).
- **Panic-DoS hardening** — `hex::decode` agent-id parsing (AAASM-4149 / 4150),
  gRPC `max_decoding_message_size`, MitM CA X.509 NameConstraints.
- **Credential scanner** extended to `gho_` / `ghu_` / `ghr_` / `github_pat_` /
  `xapp-` / `xoxr-` / `ASIA` prefixes; **proxy** decompress-then-scan by
  `Content-Encoding`.
- **`enforcement_mode` self-registration downgrade** ignored (AAASM-4121);
  agent-scoped controls bound to the token-derived id; **capability stage**
  enforced on primary + single-file paths (AAASM-4120 / 4123).
- **Dashboard** auth moved to `sessionStorage` via a `tokenStorage` helper,
  identity claim rendered (not the raw token), strict CSP, legacy token purge.
- **`file_delete` capability** governance (Epic AAASM-4092) — `FileMode::Delete`
  → `Capability::FileDelete`, allow-write-deny-delete example, RBAC-gated
  mutating dashboard controls.
