## v0.0.1-alpha.6 â€” recovers the 6 crates blocked by alpha-5

Sixth pre-release in the v0.0.1 dry-run series. Re-runs the entire
release pipeline now that the two alpha-5 blockers have been merged
to master.

### Context â€” why a fresh bump rather than recovering alpha-5

alpha-5 published successfully to 5 of 6 channels (GH Release,
Homebrew, npm, PyPI, ghcr.io). On crates.io, only 3 of the 9
publishable crates landed (`aa-core`, `aa-proto`, `aa-ebpf-common`).
The remaining 6 crates (`aa-ebpf`, `aa-runtime`, `aa-proxy`,
`aa-sandbox`, `aa-gateway`, `aa-cli`) were blocked because
`cargo workspaces publish` runs `cargo publish --verify` before
upload, and `aa-ebpf`'s `build.rs` renames a staged
`Cargo.toml.embedded` â†’ `Cargo.toml` inside the extracted-tarball
build directory. Cargo's source-mutation guard refuses the publish.

`gh run rerun --failed` uses the workflow definition that existed at
the time of the original tag push (pre-2463 fix), so re-running
cannot pick up the post-merged improvement. Bumping to alpha-6 with
a fresh tag is the cleanest way to validate the full release flow
with every recovery fix in place. On alpha-6, all 9 crates publish to
crates.io including `aa-core`, `aa-proto`, and `aa-ebpf-common` at a
fresh `0.0.1-alpha.6` (the existing alpha-5 rows remain on the
registry alongside).

### Recovery fixes verified by this tag

- **AAASM-2463 commit 1 (PR #871)** â€” `release.yml` now passes
  `--no-verify` to `cargo workspaces publish` so the publish step
  does not trip on the `cargo publish --verify` source-mutation
  guard. The actual uploaded tarball is unchanged
  (`_embedded/aa-ebpf-probes/` keeps its `.embedded`-suffixed
  manifest); pre-tag CI already validates the workspace builds
  cleanly, so the per-crate verify step is redundant.
- **AAASM-2463 commit 2 (PR #871)** â€” removed the `smoke-test:` job
  from `release.yml`. The job was declared at the same `needs:`
  level as `publish-crates`, so it ran in parallel with the publish
  steps and racing publishes / homebrew tap merges. It is removed
  for this cycle; re-introducing it correctly ordered (or as a
  separate `workflow_run`) is a future follow-up.

### Behaviour delta on the crates.io `aasm` binary

Unchanged from alpha-5. The published `aasm` binary omits the
`aasm run <tool>` and `aasm tools` subcommands while the dev-tool
subsystem is being finished. Local source builds
(`cargo build -p aa-cli`) expose the full surface unchanged. See
`docs/src/compatibility.md` for the restoration recipe.
