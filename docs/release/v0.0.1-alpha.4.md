## v0.0.1-alpha.4 — first crates.io publish

Fourth pre-release in the v0.0.1 dry-run series. The headline change:
**`cargo install aasm` now works for the first time.** Three
release-infra fixes verified by this tag.

### Fixes verified by this tag

- **AAASM-2340** — workspace published to crates.io topologically via
  cargo-workspaces; sibling content bundled into crate tarballs through
  `_embedded/` mirrors so `cargo install aasm` ships the full product
  (dashboard SPA + gRPC proto + BPF probe source). New `aasm sandbox
  run` / `aasm sandbox info` subcommands expose the WASI tool-execution
  sandbox to OSS users.
- **AAASM-2339** — `smoke-curl-installer` channel gated with `if:
  false` until `get.agent-assembly.io` is provisioned.
- **AAASM-2336** + companion node-sdk#66 + python-sdk#73 — `release.yml`
  fires `repository_dispatch` to node-sdk and python-sdk after the GH
  Release object is created; node-sdk and python-sdk consume the
  dispatch and drop their retry / silent-warning workarounds.

### CI performance follow-up (AAASM-2340)

Test job ~60 min → ~9 min. Coverage ~60 min+ → ~18 min. SonarCloud
failing → ~22 min SUCCESS. Both Integration tests jobs (ubuntu +
macOS) ~20 min timeout → ~10–15 min. Achieved by pre-building `aasm`
once and exposing it to nextest via `AASM_BIN_PATH`, eliminating
per-test `cargo run` overhead.


### Behaviour delta on the crates.io `aasm` binary

The published `aasm` binary omits the `aasm run <tool>` and `aasm
tools` subcommands while the dev-tool subsystem is being finished
(the three `aa-devtool*` crates are held back from this alpha). Local
source builds (`cargo build -p aa-cli`) expose the full surface
unchanged. See `docs/src/compatibility.md` for the restoration
recipe.
