## v0.0.1-alpha.8 — clears the 4th publish-crates blocker

Eighth pre-release in the v0.0.1 dry-run series. Re-runs the full
release pipeline with the AAASM-2797 storage-crate path-dep version
fix baked into master.

### Context — why a fresh bump rather than recovering alpha-7

alpha-7 published only `aa-core@0.0.1-alpha.7` to crates.io and then
`publish-crates` died:

```
error: failed to verify manifest at aa-storage/Cargo.toml
Caused by:
  all dependencies must have a version requirement specified when publishing.
```

cargo refuses to publish a crate that has path-deps without a
`version = "..."` literal. 5 publishable storage/cache crates added
between alpha-3 and alpha-5 (aa-storage, aa-storage-memory,
aa-storage-redis, aa-storage-sqlite-buffer, aa-cache) carried this
latent bug — the alpha-3/4/5/6 publishes all died at earlier
blockers, so it never surfaced. alpha-7 finally got past those and
exposed this next layer.

`gh run rerun --failed` uses the workflow definition at the time of
the original tag push (pre-AAASM-2797 fix). Bumping to alpha-8 with
a fresh tag validates the entire release flow end-to-end.

### Recovery fix verified by this tag

- **AAASM-2797 (PR #1024)** — Added `version = "0.0.1-alpha.7"` to 8
  path-dep declarations across 5 storage/cache crates. Pattern
  matches the existing publishable workspace crates.

### Expected post-tag sequence

1. `release.yml` fires → builds binaries → creates GH Release →
   `publish-crates` succeeds for **all 14 publishable crates** for
   the first time ever end-to-end:

   - 9 historical: `aa-core`, `aa-proto`, `aa-ebpf-common`, `aa-ebpf`, `aa-runtime`, `aa-proxy`, `aa-sandbox`, `aa-gateway`, `aa-cli`
   - 5 storage/cache (post-AAASM-2797): `aa-storage`, `aa-storage-memory`, `aa-storage-redis`, `aa-storage-sqlite-buffer`, `aa-cache`

2. `docker.yml` fires → builds + pushes ghcr.io images.
3. `notify-downstream` → `repository_dispatch` to node-sdk + python-sdk.
4. node-sdk publishes `@agent-assembly/sdk@0.0.1-alpha.8` + 4 sub-packages to npm. If `DOCS_BOT_PAT` is provisioned, the docs-version PR auto-opens AND auto-merges via AAASM-2794. Otherwise it auto-opens and waits for manual merge.
5. python-sdk publishes `agent-assembly==0.0.1a8` to PyPI.
6. `update-homebrew-tap` opens a tap PR. **Note**: the `brew install + test (macOS)` CI gate on the tap PR may still fail with the silent sandbox-exec SIGKILL — that investigation is a separate follow-up ticket. The formula itself is correct; users can install manually with `brew install`.
7. `release-status-aggregator` posts the per-channel verdict comment.

### Install

```bash
# Native binaries
brew install ai-agent-assembly/homebrew-agent-assembly/aasm  # may need --force to bypass sandbox-exec failure
curl -L https://github.com/ai-agent-assembly/agent-assembly/releases/download/v0.0.1-alpha.8/aasm-aarch64-apple-darwin.tar.gz | tar xz

# crates.io — first end-to-end validated publish of all 14 crates ever
cargo install aasm --version 0.0.1-alpha.8

# Container images
docker pull ghcr.io/ai-agent-assembly/aa-runtime:v0.0.1-alpha.8
docker pull ghcr.io/ai-agent-assembly/python:3.14-slim

# Language SDKs
pip install --pre agent-assembly==0.0.1a8
npm install @agent-assembly/sdk@0.0.1-alpha.8
go get github.com/ai-agent-assembly/go-sdk@v0.0.1-alpha.5  # go-sdk owns its own trajectory
```

### Behaviour delta on the crates.io `aasm` binary

Unchanged from alpha-5 / alpha-6 / alpha-7. The published `aasm`
binary omits the `aasm run <tool>` and `aasm tools` subcommands
while the dev-tool subsystem is being finished. Local source builds
(`cargo build -p aa-cli`) expose the full surface unchanged.
